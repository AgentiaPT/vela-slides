# Contributing to Vela Slides

Thanks for your interest in contributing to Vela Slides! This document explains how the codebase is organized and how to make changes safely.

## Development Setup

Requirements:
- Python 3.8+ (for build scripts — stdlib only, no pip packages)
- A Claude.ai account (for testing artifacts)

No Node.js, npm, or build toolchain needed.

## How to Make Changes

### 1. Identify the right part-file

See [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md) for the full map. Quick reference:

| Changing... | Edit this file |
|---|---|
| Block rendering (how slides look) | `skills/vela-slides/app/parts/part-blocks.jsx` |
| AI behavior, system prompts | `skills/vela-slides/app/parts/part-engine.jsx` |
| PDF/markdown export | `skills/vela-slides/app/parts/part-pdf.jsx` |
| State management | `skills/vela-slides/app/parts/part-reducer.jsx` |
| Import/export, sanitization | `skills/vela-slides/app/parts/part-imports.jsx` |
| Slide panel, presenter mode | `skills/vela-slides/app/parts/part-slides.jsx` |
| Chat panel | `skills/vela-slides/app/parts/part-chat.jsx` |
| Top-level app shell | `skills/vela-slides/app/parts/part-app.jsx` |

### 2. Edit the part-file directly

Never edit `vela.jsx` by hand — it's auto-generated.

### 3. Rebuild the monolith

```bash
python3 skills/vela-slides/scripts/concat.py skills/vela-slides/app/parts skills/vela-slides/app/vela.jsx
```

### 4. Test

```bash
# Run the full test suite (161 tests)
python3 tests/test_vela.py

# Run server tests (91 tests)
python3 -m unittest tests.test_serve

# Validate a deck
python3 skills/vela-slides/scripts/validate.py examples/starter-deck.json

# Assemble a test artifact
python3 skills/vela-slides/scripts/assemble.py examples/starter-deck.json --template skills/vela-slides/app/vela.jsx
```

Then paste the output `.jsx` into a Claude.ai artifact to verify.

### 5. Version bump

Increment `VELA_VERSION` in `part-imports.jsx` and add a `VELA_CHANGELOG` entry. Every change = version bump.

## Code Style

- Single-file output constraint: no ES module imports between files
- All functions must be defined before use (concatenation order matters)
- Use `const` over `let` where possible
- Inline styles over CSS classes (artifact sandbox limitation)
- SVG in blocks must use theme tokens (`{{accent}}`, `{{color}}`, `{{muted}}`, `{{bg}}`)

## Security

All contributions must maintain the security posture documented in [docs/SECURITY.md](docs/SECURITY.md):

- SVG sanitization must not be weakened
- No `eval()`, `new Function()`, or dynamic script injection
- All imported strings must pass through `sanitizeString()` or equivalent
- No credentials, API keys, or personal data in the codebase

## Pull Requests

1. Fork the repo and create a feature branch
2. Make your changes in the appropriate part-file(s)
3. Rebuild the monolith with `concat.py`
4. Test with at least one deck assembly
5. Open a PR with a clear description of what changed and why

### AI-Generated Code Disclosure

If your contribution was partially or fully generated by AI (Claude, Copilot, GPT, etc.), **include in the PR description**:

- The original prompt(s) used to generate the code
- Any spec, requirements, or reference material provided to the AI
- Which parts are AI-generated vs. hand-written
- The AI tool and model used (e.g. "Claude Opus 4, via Claude.ai")

This isn't a barrier — it's transparency. AI-assisted contributions are welcome. We just want an honest record of how code was produced, for IP provenance and for anyone reviewing the PR to understand the intent behind the output.

## Reporting Issues

Open a GitHub issue with:
- What you expected to happen
- What actually happened
- Steps to reproduce
- Browser + device info (if visual bug)

For security vulnerabilities, see [docs/SECURITY.md](docs/SECURITY.md#reporting-vulnerabilities).

### How to sign off

Add a `Signed-off-by` line to every commit in your PR:

```
Signed-off-by: Your Name <your@email.com>
```

Git makes this easy: `git commit --signoff` (or `-s`).

The `Signed-off-by` line confirms you have read and agree to the CLA above. PRs without sign-off will not be merged.
